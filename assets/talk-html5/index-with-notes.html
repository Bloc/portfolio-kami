Why talk about HTML?

  1. Because we don't enough.
  2. It's the foundation of the internet interface.
  3. It means something.
- Ctrl-Click Cast w/ Susan Robertson. - Easy these days to feel excited or compelled to learn new, hot frameworks and tools. - At expense of putting in time mastering the basics. - We can use whatever new JS framework we want, but if the DOM is meaningless then we've not done our job. - Frequencies of new hot frameworks and tools leads to ignoring fundementals.

Why use semantic HTML?

  • No new presentational value
  • More to think about in markup
  • Dramatically improved accessibility
  • Better CSS selectibility
  • SEO value
    … follow Google’s time-honored guidelines: write valid, cross-browser, accessible HTML, don’t misuse markup or “cloak” with CSS, make a site with a clear hierarchy and text links, and write good content.
    Bruce Lawson, HTML5 Doctor
- They improve accessibility dramatically without any additional efforts (e.g., roles, aria). - Easier to select in CSS. - Personally feels nicer to select elements. - SEO value? - W/ some, easier for web crawlers to pull out information (e.g., nav) - CITE: Bruce Lawson, writing for HTML5 Doctor

Semantic Thoughts

Presentational elements

<p>, <blockquote>, <code>, <strong>, <h1-6>

Structural elements

<div>
vs.
<aside>, <main>, <article>, et al.

- **"Semantic":** relating to meaning in language or logic. - Some semantic elements are fairly obvious, b/c they have presentational meaning as well. - Many of these provide clarity during development as well. - Why not just use divs? - They lack any new presentational value by default.

Semantic Thoughts

  1. Not to be used only once.
  2. Context matters. What's the parent?
  3. Location matters little.
- Many can be used multiple times, but their meaning is in relation to their context. - FOOTER could be at the top if it makes sense for the page - As I was researching this, I was regularly struck by things I now see I was doing wrong. If you experience this as well, you’re likely in good company. - Let's meet the elements! - I'm going to go alphabetically rather than in some DOM order. - Making the point that headers aren't only at the top of the page, etc.

<article>:
RSS fodder

The <article> element represents an independent item section of content.

Default role: Article

  • magazine/newspaper/blog article
  • blog post teaser
  • static pages
  • forum post
  • user-submitted comment
- Anything that theoretically could make sense on its own. - One of the more clear elements, especially for actual articles. - Also comments, forum posts. (multiple "article" on one page) - HTML5 Doctor: "Within the world of HTML markup, we can’t be held responsible for what people fill the spaces with."

<aside>:
Not just for sidebars anymore (or ever)

The <aside> element represents a section of a page that consists of content that is tangentially related to the content around the aside element…

Default role: Complementary

  • pull quotes
  • sidebars
  • advertising
  • group of in-page nav
- Never actually meant for sidebars, or content secondary to the whole site. - Now okay when not inside an article tag. - CONTEXT: It's complementary to it's parent. - Removing these would have no real impact on reading the primary content.

<footer>:
Wrap it up, b

The footer element represents a footer for its nearest ancestor sectioning content or sectioning root element.

Default role: ContentInfo

  • authorship
  • copyright info
  • limited navigation
  • appendix
  • NOT a sectioning element
- "Typically contains information about its section such as who wrote it, links to related documents, copyright data..." - Not for a new section (wouldn't have own header, footer, etc.) - Not required to be at the end, but usually is.

<header>:
Introducing... the section

The header element represents introductory content for its nearest ancestor sectioning content or sectioning root element.

Default role: Banner (if in <body>)

  • page banner
  • article headings and bylines
  • navigation
  • NOT a sectioning element
- Awkwardly similar to HEAD - primary places to set a page or section’s headings and subheadings - RULE: It should contain a heading - Probably not every time you use a heading (e.g., article SECTIONs)

<main>:
the Highlander of HTML

The main content of the body of a document or application. [It] is unique to that document and excludes content that is repeated across a set of documents.

Default role: Main

  • unique to that document or page
  • excludes repeated content (e.g., site sidebar)
  • "Must not" be a descendent of <article>, <aside>, <footer>, <header>, <nav>
  • NOT a sectioning element
- HTML5Doc: "designed to show screenreaders and assistive tech exactly where main content begins" - Use to let screen readers skip down, zoom in on mobile, promote in print.css - b/c exclusion of repeated content, written differently in different page template files.

<nav>:
for nav, but not all nav

The nav element represents a section of a page that links to other pages or to parts within the page.

Default role: Navigation

  • primarily for "major navigation", per spec
  • top nav, sub-nav
  • in-page nav to document sections
  • breadcrumbs(?)
  • not necessarily footer links
  • not aside reference links
- somewhat vague due to subjective "major navigation" definition. - Also NOT a sectioning element

<section>:
div's more interesting cousin

The section element represents a generic section of a document or application.

Default role: Region

  • like div, limited semantic meaning
  • thematically relates contents
  • a clear branch of the page's outline
  • could start with an <h1-6>
- fortunate this is the last alphabetically as it has least specific meaning - Imagine parts of a design agency's single page scrolling site for Work, Team, About Us. - Not only presentationally related, but **thematically**. - RULE: It should start with a heading

<div>:
what is it good for?
(means absolutely nothing)

The div element has no special meaning at all.

Default role: none.

  • represents its children
  • friend of complex designs
- SECTION spec: "When an element is needed only for styling purposes or as a convenience for scripting, authors are encouraged to use the div element instead." - If your site needed no design other than browser defaults, there'd be no need. - unlike semantic elements

The choir

  • article
  • aside
  • footer
  • header
  • main
  • nav
  • section

Backwards compatibility

Full support in most modern browsers, partial in Opera Mini and IE 9-11 (no main semantic meaning).

For IE <=8:

  • document.createElement("section"); and display: block; OR
  • html5shiv.js by Remy Sharp.

Rapid fire

blockquote: sourced content, not indentation

strong and em for emphasis, not styling. <i> for tone change.

<h1-6> for true headings, not styling.

If it walks and talks like a button, use <button>

- an h3 after an h2 is considered part of the h2's context

Rapid fire

<address> is only those for the page/doc/site.

There is a <menu>, but it's not the nav menu

<hr> is a "paragraph-level thematic break"

- there are a bunch more, but these are some of the most often used.

  • We call it a language, so we should treat it like one.
  • Client content markup won't be ideal.
  • CSS layout specs will mean even fewer divs.
- semicolons work to separate written ideas, but we use periods because they have meaning.
